# 客户使用体验反馈分析

**Skill 标识**: `customer-experience-feedback-analysis-v2`

其他 skill 通过 `customer-experience-feedback-analysis-v2` 引用本 skill。

## Purpose

把本地客户访谈、工单、售后反馈、客户成功记录、回访纪要、群聊反馈、问题单导出或本地材料文件夹，整理成客户使用体验反馈分析报告，不直接生成完整 PRD。

需求线索只能来自材料证据。没有客户原话、工单描述、售后记录或客户成功记录支撑的结论，必须标为 `推断` 或 `未体现`。

最终交付必须是中文 Markdown 报告文件。不要只在聊天里回答。聊天回复只给报告路径、核心发现和材料限制。

## When To Use

- 用户提供单个或多个本地文件路径。
- 用户提供一个本地材料文件夹路径。
- 材料来自客户访谈、工单、售后反馈、客户成功记录、回访纪要、群聊反馈、问题单导出、表格导出等。
- 用户希望识别使用场景、痛点、阻塞、共性诉求、影响客户数、客户名单、证据原话、影响、根因和后续 PRD 写作所需的需求线索。

不要用于外部链接、网页链接、在线表格链接、飞书/微信文档、工单系统在线地址或 CRM 在线地址导入。遇到这类输入时，停止分析，要求用户先导出为本地 `.xlsx`、`.csv`、`.md`、`.txt`、`.docx` 或 `.pdf` 文件后再提供路径。不要尝试浏览器登录、网页抓取、接口调用或在线系统取数。

## Inputs

最小输入是本地文件路径或本地文件夹路径。缺失元信息不阻塞分析，但要在报告中标记 `未提供`。

如果用户问如何提供材料，给出这个最小模板：

```text
客户名称：
产品/模块：
材料来源：访谈 / 工单 / 售后 / 客成 / 混合
时间范围：
分析目标：整理使用体验反馈，并提炼需求线索

本地文件或文件夹路径如下：
...
```

如果材料是多个客户混在一起且用户没有说明是否要合并分析，先问：`这次是按单个客户分析，还是做多客户共性分析？`

## Workflow

1. 建立材料清单：记录来源类型、文件名或输入来源、客户名称、产品模块、时间范围、内容概况。
2. 区分材料类型：访谈偏场景和期望，工单偏问题和影响，售后偏处理过程，客户成功记录偏持续使用状态。
3. 提取使用场景：客户在什么业务流程、角色、环境下使用产品。
4. 提取体验反馈：问题、阻塞、困惑、不满、正向反馈、绕行方案、客户期待。
5. 保留证据：优先引用客户原话或原始记录；不要把我方解释当成客户反馈。
6. 做问题归类：功能缺失、流程复杂、认知成本、性能稳定性、权限配置、数据可见性、集成适配、服务交付、其他。
7. 聚合共性诉求：合并表达不同但指向同一客户问题的反馈，列出影响客户数、客户名单、代表证据和涉及记录；客户名称不明确时标为 `未具名客户/区域反馈`，不要把同一客户的重复记录当成多个客户。
8. 提炼需求线索：只输出能被证据支撑的改进方向，不补写完整 PRD。
9. 写出报告：保存为 `.md` 文件；文件名使用 `客户使用体验反馈分析报告_v1.md`，如已存在则递增版本号。

## Output Contract

报告使用以下结构，按材料复杂度精简或展开：

1. 分析结论摘要
2. 材料清单与覆盖范围
3. 客户使用场景
4. 客户体验反馈明细
5. 共性诉求与影响客户
6. 证据原话与事实/推断区分
7. 问题归类与影响判断
8. 需求线索
9. 不建议直接转需求的内容
10. 信息缺口与后续追问

`客户体验反馈明细` 表格建议包含：场景、反馈类型、客户问题、影响、客户当前处理方式、证据、事实类型。

`共性诉求与影响客户` 表格建议包含：共性诉求、对应问题、影响客户数、客户名单、代表证据、涉及记录、需求判断。

`需求线索` 表格建议包含：需求线索、对应体验问题、影响客户数、客户名单、目标用户、建议方向、证据强度、优先级建议、PRD 可用输入。

聊天收尾包含：

- Markdown 报告路径。
- 3 条以内核心发现。
- 材料覆盖限制或需要补充的信息。

## Evidence Standards

- `事实`：材料中直接出现的客户说法、工单描述或记录。
- `推断`：由多条材料共同推出，但没有直接原话。
- `未体现`：材料没有可靠依据。
- 不把重复出现于同一条工单或同一次访谈的内容当成多客户共性。
- 不把一次性服务协调、排期、人工处理直接转成产品需求，除非它暴露了可产品化的问题。
- 不输出泛泛的“优化体验”“提升效率”；必须说明具体场景和证据。

## Acceptance Criteria

- 输入只来自本地文件或本地文件夹；遇到外部链接或在线系统地址时，已停止并要求用户导出本地文件。
- 多客户材料合并方式不明确时，已先确认按单客户分析还是多客户共性分析。
- 已生成中文 Markdown 报告文件；同名文件存在时已递增版本号，没有覆盖旧报告。
- 报告包含材料清单与覆盖范围，并标记缺失元信息为 `未提供`。
- 使用场景、体验反馈、共性诉求、问题归类和需求线索均来自材料证据。
- 无直接证据的结论已标为 `推断` 或 `未体现`，没有把我方解释当成客户反馈。
- 影响客户数和客户名单没有把同一客户的重复记录重复计数；客户名称不明确时使用 `未具名客户/区域反馈`。
- 需求线索没有写成完整 PRD，也没有输出泛泛的“优化体验”“提升效率”。
- 一次性服务协调、排期、人工处理未被直接转成产品需求，除非报告说明了对应可产品化问题。
- 聊天收尾包含 Markdown 报告路径、3 条以内核心发现、材料覆盖限制或需要补充的信息。
