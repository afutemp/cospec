# TR1 用户需求说明书生成器

**Skill 标识**: `tr1-requirements-spec`

其他 skill 通过 `tr1-requirements-spec` 引用本 skill。

## Purpose

将 DCP2-1 立项材料、客户诉求、Demo/原型、竞品分析、共创验证、用户旅程和技术评估结论，转换成新版 TR1 用户需求材料。

新版 TR1 不再默认生成一份大而全的旧式详细规格，而是按需求规模拆成：

- **大需求评审版**：给人评审，讲清楚当前版本为什么做、做什么、价值是什么、方案如何验证、风险是否可控。
- **大需求 AI 上下文版**：给 AI 生成系统需求、EPIC/Feature/Story、Tech、测试和编码任务。
- **小需求评审版**：给单点功能、局部优化、小范围客户诉求快速评审。

评审版用于促成共识，不追求系统规格完整性；AI 上下文版用于支撑后续生成，不写成散文。

## When To Use

当用户提出以下任务时使用本 Skill：

- 生成 TR1 用户需求说明书。
- 从 DCP2-1 材料生成用户需求。
- 编写大需求评审版。
- 生成大需求 AI 上下文版。
- 编写小需求评审材料。
- 把立项材料、客户拜访、Demo、竞品分析、共创验证、用户旅程或技术验证结果转成 TR1 评审材料。
- 按大/小需求选择模板。
- 为后续 Story、Tech 或编码任务准备 AI 上下文。
- 将太长的 TR1 拆成人看的评审版和 AI 看的上下文版。

不要在用户只要求 Playbook、EPIC、Feature、Story、Tech 或代码任务时默认生成完整 TR1；此时只按本次目标产物裁剪相关内容。

## Extension Points

Before starting, read `cospec.config.json` from the plugin root (2 levels above this skill's base directory). No fallback needed — the config always has valid defaults.

| Config Field | Purpose |
|---|---|
| `config.templates["user-requirement"]` | Default user requirement (TR1) template for lightweight usage |
| `config.templates["tr1-large-review"]` | Large-requirement review edition template |
| `config.templates["tr1-large-ai"]` | Large-requirement AI-context edition template |
| `config.templates["tr1-small-review"]` | Small-requirement review edition template |
| `config.rules["requirement-checklists"]` | Requirement checklists directory used for self-review |
| `config.evaluators["tr1-requirements-spec"]` | Evaluator skill name after this stage; `false` = disable gate |

## Inputs

优先从用户提供的材料中提取，不要先让用户手填。常见输入包括：

1. DCP2-1：战略目标、主打价值、客群、大方向。
2. 竞品分析：竞品能力、差距、替代方案、竞争策略。
3. 过往产品差距和好口碑：历史短板、客户认可点、不能破坏的体验。
4. 共创与验证内容：Demo 初稿、客户拜访、共创结论、技术讨论。
5. 用户旅程与技术验证：用户旅程设计、流程、技术验证结果。
6. 客户本次输入：本次客户原始诉求、反馈、约束和新增要求。

也可接收局部输入，例如单个场景、单个客户诉求、单个功能点、单条 Playbook、指定 Feature 或 Story。

## Workflow

### 1. 识别本次任务目的和需求范围

生成前先完成“双重裁剪”：先按本次任务目的裁剪，再按本次需求范围/场景裁剪。本次 prompt 的目标和范围优先于输入材料完整度；完整材料只是材料池，不等于本次输出范围。

先识别：

1. **本次目标产物**：TR1 评审版 / TR1 AI 上下文版 / 大需求双文档 / 小需求评审版 / Playbook / EPIC / Feature / Story / Tech / 代码任务。
2. **本次需求范围**：版本全量 / 单个场景 / 单个客户诉求 / 单个功能点 / 单条 Playbook / 指定 Feature 或 Story。
3. **本次抽取范围**：只抽取同时匹配“目标产物”和“需求范围”的材料。

执行前必须先在对话中输出简短判断：

```text
本次目标产物：[填写]
本次需求范围：[版本全量/单个场景/单个客户诉求/单个功能点/单条 Playbook/指定 Feature]
本次抽取范围：[说明只抽取哪些材料]
本次不覆盖范围：[无关场景/无关功能/无关客户诉求，或“待补充”]
```

行为规则：

- 如果输入材料完整，但本次 prompt 只指定一个场景、客户诉求、功能点或 Playbook，禁止默认生成完整版本 TR1。
- 如果用户说“生成本版本 TR1”，且没有指定局部场景，按版本全量处理。
- 如果用户说“生成 TR1”，但材料里存在多个明显场景且本次范围不清晰，先给出候选范围并询问一个最小问题；用户不补充时，按“范围待确认”生成初稿。
- 无关场景最多放入“本次不覆盖范围”或不出现，不进入需求结论、功能列表、风险结论、验收标准和 AI 上下文 ID 链。

### 2. 执行阻塞式材料确认

生成 TR1 大需求或小需求前，必须先做一次**用户可见的执行前提示**，再生成文件或正式正文。这个提示不能只写进最终文档里，必须先在对话中输出。

执行前提示是 TR1 大需求和小需求共同的阻塞式交互门。Playbook、EPIC、Feature、Story、Tech 或代码任务按目标产物裁剪规则处理，不强制进入该阻塞式材料确认。

检查 6 类输入材料：

| 材料类别 | 检查内容 | 状态 |
|----------|----------|------|
| DCP2-1 | 战略目标、主打价值、客群、大方向 | 已提供/部分提供/未提供/用户选择不提交/空材料 |
| 竞品分析 | 竞品能力、差距、替代方案、竞争策略 | 已提供/部分提供/未提供/用户选择不提交/空材料 |
| 过往产品差距和好口碑 | 历史短板、客户认可点、不能破坏的体验 | 已提供/部分提供/未提供/用户选择不提交/空材料 |
| 共创与验证内容 | Demo 初稿、客户拜访、共创结论、技术讨论 | 已提供/部分提供/未提供/用户选择不提交/空材料 |
| 用户旅程与技术验证 | 用户旅程设计、流程、技术验证结果 | 已提供/部分提供/未提供/用户选择不提交/空材料 |
| 客户本次输入 | 本次客户原始诉求、反馈、约束和新增要求 | 已提供/部分提供/未提供/用户选择不提交/空材料 |

如果本次目标产物是**小需求评审版**，还必须单独确认 Demo 适用性：

| 确认项 | 状态 | 处理方式 |
|--------|------|----------|
| Demo 是否适用 | 已有 Demo/需要但未提供/不需要 Demo/待确认 | 有 Demo 则说明证明点；不需要 Demo 则说明原因和替代验证方式；需要但未提供才写“待补充”或进入 `OPEN-*` |

阻塞交互、非强制提交含义：

- 如果用户补充材料，按补充后的材料继续。
- 如果用户没有材料或选择不提交，必须等待用户明确回复“继续”“无补充”或“跳过”后，再基于已有材料生成初稿。
- 用户不提交或材料为空不视为错误；普通缺失写“待补充”。
- 在收到用户明确补充或跳过回复前，不生成文件或正式正文。
- 只有产品/版本/需求对象完全无法判断时，才先问最少必要问题。
- 影响范围、方案选择、技术风险、评审决策或后续编码的关键未确认项进入 `OPEN-*`。

### 3. 判断大需求或小需求

仅当本次目标产物是 TR1 类文档时执行本阶段；如果本次目标产物是 Playbook、EPIC、Feature、Story、Tech 或代码任务，按目标产物裁剪规则直接处理，不进入 TR1 大/小需求判定。

判定规则：

| 判定维度 | 大需求 | 小需求 |
|----------|--------|--------|
| 影响范围 | 跨模块、跨部门、跨产品线或影响多个客户角色 | 单模块、单页面、单流程或局部客户诉求 |
| 价值属性 | 承接版本战略、关键价值点或竞争优势设计 | 解决明确痛点，但不一定承接重大版本战略 |
| 场景复杂度 | 多个关键场景、多个 Playbook 或完整端到端旅程 | 单一场景，可通过 Demo、截图、测试、数据或评审结论直接说明 |
| 交付复杂度 | 有关键技术风险、跨团队依赖或专项验证 | 依赖少、风险可控、无需多部门专项协同 |
| AI 上下文需要 | 默认需要独立 AI 上下文版 | 默认不需要，除非用户明确要进入 AI 编码 |

默认选择：

- 满足任一大需求条件，按大需求处理。
- 明确是单点优化、客户小诉求或局部体验改进，按小需求处理。
- 用户指定模板或明确说“大需求/小需求”时，以用户指定为准。

### 4. 按场景相关性抽取材料

按以下顺序判断材料是否属于本次范围：

1. 优先匹配用户本次 prompt 明确提到的场景名、客户名、客户角色、功能名、Playbook 名、Feature/Story ID。
2. 其次匹配材料里的 Demo 页面、原型节点、用户旅程步骤、客户原话、痛点、技术验证结论和异常路径。
3. 只保留能与本次场景/需求切片建立明确关系的 DCP2-1、竞品分析、差距口碑、共创验证、旅程、技术验证和客户输入。
4. 不能建立关系的材料不强行使用；如果它可能影响本次范围但证据不足，放入“待补充”或 `OPEN-*`。

材料使用规则：

- **客户本次输入**是一手证据，优先作为本次场景/需求切片的原始诉求、主要痛点、范围依据和本期覆盖判断。
- **DCP2-1**只取与本次场景相关的战略目标、目标客群、主打价值、版本目标和范围边界，不覆盖客户原话。
- **竞品分析**只取与本次场景相关的竞品差距、替代方案、差异化点和方案策略，不直接生成需求承诺。
- **过往产品差距**只取影响本次场景的痛点、质量问题、方案改进和风险；**好口碑**只取本次场景需要保留的能力、设计原则和不可破坏项。
- **共创、Demo、客户拜访、技术讨论**只取本次场景相关的 Demo、拜访反馈、共创结论、技术讨论、验证结论、风险、待补充和 `OPEN-*`。对小需求，先判断 Demo 是否适用；Demo 不适用时，改用截图、测试、数据、日志或评审结论作为验证材料。
- **用户旅程设计**只取本次场景对应的主流程、分支流程、异常路径和操作步骤；**技术验证结果**只取本次场景相关的技术评估、风险、验收和测试上下文。

### 5. 生成对应产物

本次任务目的优先于默认输出策略。用户明确只要求评审版、AI 上下文版、小需求、Playbook、EPIC、Feature、Story、Tech 或代码任务时，只生成指定产物，不补生成完整链路。

目标产物裁剪规则：

| 本次目标产物 | 优先抽取内容 | 不应默认生成 |
|--------------|--------------|--------------|
| TR1 评审版 | 本次范围内的结论、价值判断、方案说明、Demo 证明点、风险是否可控 | 完整 AI 上下文、EPIC/Feature/Story 细节、编码任务 |
| TR1 AI 上下文版 | 本次范围内的来源 ID、规则、边界、异常、对象、接口、测试、`OPEN-*` | 面向人工评审的长篇叙事 |
| 大需求双文档 | 本次范围内的评审版 + AI 上下文版 | 未被本次范围覆盖的其他场景 |
| 小需求评审版 | 单点背景、痛点、原始诉求、功能需求、验收、影响范围、依赖风险 | 战略承接、完整竞品、完整 Playbook、完整 AI 上下文 |
| Playbook | 场景、角色、痛点、Demo、旅程步骤、配置、验证点、必要风险 | TR1 全套、其他 Playbook、EPIC/Feature/Story 全量拆解 |
| EPIC | `VAL/REQ/EPIC/PB/OPEN` 中与本次范围相关的价值和范围 | Feature/Story/Tech 细节 |
| Feature | `FEAT/BR/AC/DEMO/NFR` 中与本次范围相关的功能边界和验收 | 无关 Feature、完整版本 TR1 |
| Story | `ST/BR/AC/ERR/PERM` 中与本次范围相关的用户故事和验收 | 无关 Story、技术实现全集 |
| Tech | `OBJ/INT/NFR/RISK/OPEN` 中与本次范围相关的技术对象、接口、约束和风险 | 业务评审叙事、无关模块 |
| 代码任务 | 实现上下文、接口、规则、验收、风险和未确认项 | 需求评审全文、无关功能代码任务 |

## Output Contract

### 大需求输出

大需求默认生成两份 Markdown：

```text
{版本或项目名}_大需求用户需求规格说明书_评审版.md
{版本或项目名}_大需求用户需求规格说明书_AI上下文版.md
```

评审版必须控制在“人能评审”的粒度，不展开细粒度系统规格。评审版只输出输入材料的结论和评审判断：

- 只写材料证明了什么、对需求有什么影响、风险是否可控。
- Demo 已存在时，只说明 Demo 证明的价值、流程或效果。
- 客户本次输入保留原始诉求摘要，不把内部方案改写成客户原话。
- 对竞品、共创、技术讨论等材料给出结论，不堆叠原始记录全文。
- 非功能性需求只写评审需要的结论、关键约束、明显风险和验证方式，不要求在 TR1 评审版完整填写 DFX 自检文档。
- DFX 自检文档作为后续 Feature/Story/Tech 的基线库引用，不回灌到 TR1 评审版主体。

评审版不要内嵌：

- 完整 EPIC/Feature/Story 层级。
- 详细接口字段、数据对象字段清单。
- 完整异常状态、弹窗、权限矩阵。
- 编码任务、Tech 拆解、测试用例全集。
- 完整 DFX 自检表或通用质量基线清单。

AI 上下文版必须是结构化上下文包，重点不是叙事，而是可追溯、可生成、可测试。必须保留：

- 来源 ID、来源类型、证据摘要和可信度判断。
- 客户原始诉求、Demo/原型细节、异常状态、规则、边界、对象、接口、测试和 `OPEN-*`。
- 客户本次输入、DCP2-1、竞品、共创验证、技术验证结果的追溯关系。

AI 上下文版必须使用稳定 ID：

```text
REQ / VAL / PB / DEMO / EPIC / FEAT / ST / BR / PERM / OBJ / INT / NFR / ERR / AC / TC / QI / OPEN
```

### 小需求输出

小需求默认只生成一份 Markdown：

```text
{需求名称}_小需求用户需求说明_评审版.md
```

小需求只保留轻量结构：

1. 背景及目标。
2. 客户主要痛点。
3. 客户原始诉求。
4. 关键功能需求：功能列表、Demo/验证材料、验收标准、影响范围、依赖与风险。
5. 附录和自检清单。

小需求不要强制写：

- 完整战略承接。
- 完整竞品分析。
- 完整 Playbook。
- 独立 AI 上下文版。

如果用户明确要求“小需求也要进入 Story/Tech/代码任务或 AI 编码”，只补充与本次小需求范围相关的轻量 AI 上下文，不生成大需求 AI 上下文版，也不扩展为完整大需求材料。

### 缺失与 OPEN 规则

- 普通内容缺失写“待补充”，例如截图路径、负责人、发布日期、证据来源等。
- Demo 只有在确认“需要但未提供”时才写“待补充”；确认“不需要 Demo”时写“不适用”并说明替代验证方式。
- 影响范围、方案选择、技术风险、评审决策或后续编码的未确认事项进入 `OPEN-*`，不得写成已确认事实。
- 关键输入完全缺失且会导致文档方向无法判断时，先说明缺口，再向用户索取最少必要信息。

## Acceptance Criteria

完成任务前逐项检查：

- 是否在生成 TR1 大需求或小需求前，输出了用户可见的执行前检查。
- 是否完成本次任务目的和本次需求范围的双重裁剪。
- 是否等待用户补充材料，或等待用户明确回复“继续”“无补充”或“跳过”后才生成 TR1 正文或文件。
- 是否完成 6 类输入材料检查，并记录用户补充/跳过结果。
- 小需求是否单独确认 Demo 是否适用，并在不需要 Demo 时给出替代验证方式。
- 是否避免无关场景进入需求结论、功能列表、风险结论、验收标准和 AI 上下文 ID 链。
- 大需求是否输出评审版和 AI 上下文版；小需求是否默认只输出轻量评审版。
- 评审版是否讲清楚当前版本为什么做、做什么、价值是什么、Demo/方案如何验证、风险是否可控。
- 评审版是否避免堆叠系统规格、编码细节、完整异常状态、权限矩阵、测试用例全集和完整 DFX 自检表。
- AI 上下文版是否建立稳定 ID 体系，并形成 `REQ -> EPIC -> FEAT -> ST -> BR -> AC` 追溯链。
- AI 上下文版是否只为本次范围内的场景或需求切片建立 ID 和追溯链。
- 是否保留客户原始诉求、真实痛点、材料来源、证据摘要和追溯关系。
- 普通内容缺失是否写为“待补充”。
- 影响范围、方案选择、技术风险、评审决策或后续编码的关键未确认事项是否进入 `OPEN-*`。
- 是否使用对应模板，并保持模板章节和表格口径。
- 是否没有把旧 V3.8 模板作为默认入口；仅在用户明确要求时使用旧模板。
