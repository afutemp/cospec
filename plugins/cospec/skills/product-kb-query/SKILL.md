# 产品知识库查询

**Skill 标识**: `product-kb-query`

其他 skill 通过 `product-kb-query` 引用本 skill。

## When To Use

当用户或上游 skill 需要了解以下主题时调用本 skill：

- 产品战略、价值定位、目标场景、版本演进、里程碑、路线图。
- 目标客户、用户角色、权限矩阵、JTBD、客户问题、用户旅程、机会地图。
- 需求池、优先级、版本范围、发布计划。
- 功能规划、需求规格、主场景、业务规则、状态模型、验收标准。
- 非功能需求、性能、容量、可靠性、安全、合规、兼容性。
- 依赖关系、风险、跨团队协作、外部系统对接、决策升级。
- 验证计划、成功指标、实验、假设、反馈、发布复盘。
- 对知识库文档的查询、检索、查找、总结、解释。

触发信号词包括：知识库、查询、检索、查找、总结、解释一下、有哪些、是什么、怎么做、风险、依赖、优先级、路线图、验收、指标。

## Extension Points

本 skill 读取 `cospec.config.json` 的 `kb` 配置来决定如何定位知识库：

| 配置项 | 含义 | 默认值 |
| :--- | :--- | :--- |
| `kb.skill` | 主 KB 查询 skill 名称。若配置为 `product-kb-query`，则启用统一注入；若为其他 skill 名称，则本 skill 通常不会被直接调用。 | `product-kb-query` |
| `kb.localPath` | 本地知识库目录（相对插件根目录或绝对路径）。**必须配置**，无默认值，不自动探测。 | - |

**Discovery order**：`KB_ROOT:` 参数（由调用方传入） → `config.kb.localPath` → 自动通过 `product-kb-server` 下载。

> 注意：当本 skill 被其他 skill 调用时，输入中已经包含一个具体的 KB 查询问题，因此不需要询问用户。如果信息不足，直接说明"当前知识库未覆盖该问题"。

## Inputs

调用本 skill 时，传入一个**具体的问题字符串**。

当由其他 skill 调用时，参数可能以 `KB_ROOT: <path>` 开头，用于显式指定知识库根目录：

```
KB_ROOT: product-kb/

请总结与当前需求相关的产品战略、目标客户、已有功能、已知风险和约束。
```

普通调用示例：

- `"请总结与当前需求相关的产品战略、目标客户、已有功能、已知风险和约束。"`
- `"认证方式有哪些？"`
- `"项目范围是什么？"`
- `"有哪些已知风险？"`

如果问题为空或过于宽泛，先尝试按目录结构做一次总览性回答。

## Workflow

### 1. 读取配置、定位并更新知识库根目录

**每次查询前都必须先尝试更新知识库**，确保本地不是旧版本。`product-kb-server download` 会自动比对服务器 `current_version_id` 和本地 `.kb-version`，一致则跳过，不一致则更新。

1. **解析传入参数**：如果参数以 `KB_ROOT:` 开头，提取该前缀后面的路径作为知识库根目录，剩余部分作为问题。例如 `KB_ROOT: product-kb/\n\n认证方式有哪些？` → 根目录 `product-kb/`，问题 `认证方式有哪些？`。`KB_ROOT:` 显式指定时跳过更新检查，直接使用该目录。
2. 如果参数中没有 `KB_ROOT:`，则读取 `<plugin-root>/cospec.config.json` 的 `kb.localPath`。
3. **尝试更新当前知识库：**
   - 如果 `kb.localPath` 已配置且目录有效、包含 `.md` 文件：
     - 从 `kb.localPath` 推断当前知识库名称：取路径的 basename（例如 `~/.cospec/kb/vdi/` → `vdi`）。
     - 调用 `<plugin-root>/skills/product-kb-server/scripts/product-kb-server.cjs download --kb <kb-name>` 触发更新检查。
     - 如果 `download` 输出包含 `is up to date ... skipping`，继续查询，不额外提示。
     - 如果 `download` 输出包含 `done` 或 `updating`，说明执行了下载/更新，在最终回答开头提示：
       > "已为你下载/更新知识库 `vdi`。以下是查询结果："
   - 如果推断的 KB 名称在服务器上不存在（download 报错），则放弃更新检查，继续使用本地现有知识库查询，并在回答中标注"未能检查知识库更新，使用本地版本"。
4. **如果 `kb.localPath` 未配置（`null`），或配置的目录不存在、或不包含任何 `.md` 文件，自动尝试通过 `product-kb-server` 下载知识库：**
   1. 调用 `<plugin-root>/skills/product-kb-server/scripts/product-kb-server.cjs list` 获取可用知识库列表（输出格式为 `NAME\tDESCRIPTION`）。
   2. 如果列表为空，返回错误："kb-server 上没有可用知识库，请联系管理员上传。"
   3. 如果列表中只有 1 个知识库，自动执行下载：
      ```bash
      <plugin-root>/skills/product-kb-server/scripts/product-kb-server.cjs download --kb <kb-name>
      ```
   4. 如果列表中有多个知识库：
      - 将当前问题、前文上下文中的关键词与各知识库的 `DESCRIPTION` 进行语义匹配，挑选最相关的一个。
      - 如果匹配到一个明显最相关的知识库，自动执行下载。
      - 如果匹配不到或不确定，返回错误："发现多个可用知识库（`vdi`, `xxx`, `yyy`），无法自动判断。请先运行 `/product-kb-server download --kb <name>` 指定要下载的知识库，或手动配置 cospec.config.json。"
   5. 下载完成后，`product-kb-server` 会自动把 `kb.localPath` 写回 `cospec.config.json`。
   6. 如果实际发生了下载/更新，在最终回答开头提示：
      > "已为你下载/更新知识库 `vdi`。以下是查询结果："

### 2. 读取检索索引

在知识库根目录下查找 `search-index.md`：

- 如果存在，**优先读取**该文件，获取目录语义、文件总览、关键词/实体索引。
- 如果不存在，则按后续步骤中的目录结构进行 fallback 检索，并提示用户可运行 `/product-kb-index` 生成索引以获得更精准的检索效果。

### 3. 按问题类型映射到优先目录

根据问题中的关键词，优先定位到以下目录（按优先级排序）：

| 问题类型 | 优先目录 | 典型文件 |
| :--- | :--- | :--- |
| 产品战略、价值、定位、目标场景 | `00-综述/` | `01-产品战略与价值.md` |
| 版本、演进、里程碑、路线图、发布计划 | `00-综述/`、`02-规划与范围/` | `02-版本演进与里程碑.md`、`02-版本范围与路线图.md` |
| 目标客户、角色、权限、JTBD、客户问题、用户旅程 | `01-用户与机会/` | `01-目标客户与角色.md`、`02-客户问题与JTBD.md`、`03-用户旅程与机会地图.md` |
| 需求池、优先级、版本范围 | `02-规划与范围/` | `01-需求池与优先级.md`、`02-版本范围与路线图.md` |
| 具体功能、需求规格、主场景、业务规则、验收标准 | `03-功能规划/` | 对应 `{功能名称}.md` |
| 性能、容量、可靠性、安全、合规、兼容性 | `04-质量与约束/` | `01-非功能需求与合规.md` |
| 依赖、风险、协作、外部系统、决策升级 | `05-协作与依赖/` | `01-依赖风险与协作.md` |
| 指标、实验、假设、反馈、验证、复盘 | `06-验证与反馈/` | `01-指标实验与反馈.md` |
| 路由、证据来源、术语、资料索引 | `附录/` | `01-原型与资料索引.md` |
| 宽泛总结、知识库总览 | 根目录 + `00-综述/` | `README.md`、`00-综述/` 下所有文件 |

### 4. 在目标目录内检索并读取文件

1. 如果已读取 `search-index.md`，先用其中的“关键词/实体索引”精确定位到 1-3 个最相关文件；
2. 否则使用 `Glob` 列出已定位目录下的 `.md` 文件，根据文件名和路径中的关键词进一步筛选，通常选择 1-3 个最相关的文件。
3. 如果问题跨多个目录（如"项目的风险和价值"），则分别从对应目录各选 1-2 个文件。
4. 使用 `Read` 读取选中的 markdown 文件。如果文档很长，优先读取标题匹配关键词的章节、表格和列表。
5. 使用 `Grep` 验证关键词确实存在于目标文件中，避免误读不相关文件。

### 5. 组织回答

1. 基于读取到的内容回答传入的问题。
2. 使用表格、列表等结构化方式呈现。
3. 明确标注信息来源，例如：`来源：03-功能规划/02-服务功能.md`。
4. 对于 `[OPEN]`、`[ASSUMPTION]`、`[BLOCKED]`、`待确认`、`待验证` 等标注，如实说明不确定性，不编造。
5. 如果知识库中确实没有相关内容，直接说明"当前知识库未覆盖该问题"。

## Output Contract

输出一个 Markdown 格式的知识库回答，包含：

1. **核心结论**：直接回答问题的 2-5 条要点。
2. **结构化细节**：必要时用表格或列表展开。
3. **来源标注**：每条事实后标注相对路径。
4. **不确定性标注**：对 `[OPEN]` / `待确认` / `待验证` 内容显式说明。

示例格式：

```markdown
根据 `03-功能规划/02-服务功能.md`：

1. 认证模块包含 14 个认证子页（密码安全、外部 LDAP/RADIUS、证书/USB Key、…）。
2. 主认证可设置为“全部满足”或“任一满足”；次认证包括硬件 ID、短信认证、动态令牌、TOTP。

> 来源：`03-功能规划/02-服务功能.md`
```

## Acceptance Criteria

- 只基于知识库内容回答，不引入外部知识。
- 不编造知识库中没有的信息。
- 保留原始标注：`【原文·界面】`、`【事实·界面】`、`【推导·待审】`、`【待验证】` 等。
- 标注不确定性：遇到 `[OPEN]`、`待确认`、`待验证` 时，明确告诉调用方该信息尚未确认。
- 引用路径：尽量使用相对路径引用来源文档。
- 被其他 skill 调用时，不主动追问用户；若信息不足，直接返回"当前知识库未覆盖该问题"。
- 当 `kb.localPath` 已配置时，每次查询前主动调用 `product-kb-server download` 检查/更新知识库；当未配置时，自动下载知识库。除此之外保持只读，不生成或修改任何文档、索引、Meta 和报告。
