# 问题求解型竞品分析

**Skill 标识**: `competitor-problem-solving`

其他 skill 通过 `competitor-problem-solving` 引用本 skill。

> 输出模板见 `references/输出模板.md`

## 厂商选择

- 只参考海外 TOP 厂商（Gartner 魔力象限领导者 / 头部客户首选品牌）
- 不参考国内厂商或仅存在竞争关系的非标杆厂商
- 终端安全示例：CrowdStrike, Defender, Palo Alto, Trend Micro, ESET

## 信息来源

**所有功能信息必须提供原始链接，否则不能作为输出内容。**

优先使用的来源（按优先级排序）：
1. 产品控制台（需登录）
2. 登录后才能获取的用户手册、白皮书
3. 官方文档站点：CrowdStrike → 控制台 + Reddit；ESET → https://help.eset.com；Defender → https://learn.microsoft.com
4. 产品官方论坛、官方发布信息
5. YouTube 等视频平台的官方产品演示和技术分享（可补充截图）

**来源链接要求**：
- 所有结论必须附带参考链接，不能只写结论不给出处
- 链接必须可访问（打开后不能是 404、空白页、域名不存在）
- 优先使用厂商官方域名，避免第三方文件托管链接
- 标注来源类型：官方文档 / 官方社区 / 第三方技术分析 / 专利文件

**正文内联标注**：
- 每个关键功能描述在首次出现时，标注来源类型（如 `[控制台]` `[PDF p.4]` `[官方文档]`）
- 格式示例：`CrowdStrike 的 ML Exclusions 和 IOA Exclusions 是完全独立的两类排除 [控制台 + PDF p.4]`
- 不要求每条重复标注同一来源，但每个功能块的第一个关键断言必须标注

禁止使用：推断臆想的内容、机构分析报告（非一手信息）、无法提供原始链接的二手资料

## 不确定信息标注

没有明确来源或来源未经打开校验的内容：
- 标注为 **【待验证】**——有线索但未确认
- 标注为 **【待确认】**——完全无依据的推断
- 禁止将推断补成确定事实。如果某家厂商的做法找不到文档验证，标注"【待验证】某厂商在此问题上的做法尚未从官方文档中确认"

## 状态机门禁

每轮只能处于一个阶段。当前阶段输出未经用户确认，不得进入下一阶段。

**Step 0：输入理解确认（最关键门禁）**
- 目标：澄清用户问题的本质——不是用户问什么就直接搜什么
- 必须完成：①拆解问题（架构问题？策略问题？交互体验问题？）②追问底层目标（表面问题 ≠ 真正要决策的点）③界定边界（哪些是业界共识？哪些是厂商定位决定的？）④如果用户带着预设方案来问（"我想做 X，看他们怎么做的"），提醒"友商可能不是通过 X 来解决问题，先回归问题本质"
- 退出条件：双方对"真正要回答的问题"达成一致，用户确认不再带预设方案
- **退出前小结**：用户确认后，用一句话回溯总结问题本质（"已确认：真正要回答的问题是 X，属于 Y 类问题，关键决策点是 Z"），再进入 Step 1
- 禁止：S0 未充分讨论就搜索答案；用户带着方案问就直接搜那个方案

**Step 1：厂商与来源确认**
- 目标：确认参考厂商范围、每家具体模块、可用来源
- 关键：问题求解型需精准匹配——不是泛泛看产品大类，而是聚焦与问题相关的具体模块。**即使用户没有新增或删减厂商，也要在进入 Step 2 前做简短确认**
- 退出条件：厂商列表 + 每家相关模块 + 每家参考来源已确认
- **退出前小结**：向用户确认"拟分析 X 家厂商（Y 模块），来源包括 A/B/C"
- 禁止：范围太大导致答非所问

**Step 2：基于资料给出解答**
- 目标：基于权威来源（控制台/用户手册/白皮书），逐厂商分析其解法，输出对比和设计意图
- **中间确认件**：先以摘要形式向用户展示：①每家关键发现 ②对比表 ③一句话结论方向 ④待闭环清单。**每家厂商至少声明 1 个信息缺口**——如果某家的设计意图只能基于产品推断而无法从官方文档验证，标注"【推断——未从官方文档找到设计决策说明】"
- 退出条件：用户确认分析方向 OK、结论方向正确
- 禁止：用推断替代文档验证；结论没有来源支撑；不经中间确认直接生成完整文档

**Step 3：验证与补充**
- 目标：向用户确认"以上是否解答了原始问题"，是否需要补充其他厂商或深入某个细节
- **强制动作**：文档输出后，必须在答复末尾显式提问"以上是否解答了您的原始问题？是否需要补充其他厂商或深入某个细节？"，这不是可选行为
- 退出条件：用户确认解答了原始问题，或明确需要补充的方向
- 禁止：输出完就走，不验证是否真正回答了用户的问题

**阶段跳跃处理**：如果用户要求跳过阶段直接生成文档，输出开头标注：`⛔ 草稿——未经阶段确认，不应直接作为最终结论`

## 中立性

**不带任何立场和假设**。目标是充分了解友商做法，不预判用户想做的方案。

## 核心约束

- 先讨论问题本质，不要用户问什么就直接搜答案——用户问的可能只是表面问题
- 如果用户带着预设方案来问（"我想做自动关闭，看他们怎么做的"），主动提醒回归问题本质
- 明确每家"为什么这么解"——产品定位 × 目标客群 × 产品与技术路线 × 收益与代价

## 对话流程

### S1 — 问题本质讨论（最关键）

**不要直接搜索答案。** 先和用户讨论：
1. 拆解问题：设计策略 / 技术架构 / 交互体验？
2. 追问底层目标：表面问题 ≠ 真正要决策的点
3. 界定边界：哪些是业界共识？哪些是厂商定位决定的？
4. 反对预设方案：用户带着方案来问时，提醒回归问题本质

**退出条件**：双方对"真正要回答的问题"达成一致。

### S2 — 确认厂商与来源
列出厂商（聚焦与问题相关的具体模块）+ 每家参考来源。询问用户是否有产品环境。
**退出条件**：厂商列表 + 来源已确认。

### S3 — 基于资料给出解答

输出四个部分：
1. **各厂商解法**：功能机制、设计逻辑、配置方式、边界限制、截图 + 链接
2. **对比矩阵**：设计策略 / 核心机制 / 适用场景 / 限制代价
3. **设计意图解读**：每家产品定位 × 目标客群 × 产品与技术路线 × 收益与代价
4. **最佳实践提炼**：业界共识、差异化选择、潜在坑点

### S4 — 验证 + 补充
询问用户是否解答了原始问题。如果不满意，回到 S1 重新澄清或扩大范围。

## 输出规范

- 结构参考 `references/输出模板.md`
- 先给出直接回答，再展开分析。不让用户读完整篇才知道结论
- 输出中回顾 S1 的问题本质讨论
- 每个厂商给出"对原始问题的回答"表
- 每家附完整设计意图解读
- 区分业界共识、差异化选择、潜在坑点
- 来源表放附录，截图内嵌
- **推论建议标注**：如果设计建议来自于跨界参考或产品推理而非友商直接做法，标注为【推论建议】——区分'友商已验证的做法'和'基于分析的推论'

## 错误处理

### 当用户指出错误时

1. **立即承认**——先确认用户质疑的具体内容，不辩解
2. **重新检索**——用更精确的关键词重新搜索被质疑的结论
3. **对照原文**——引用原始文档的具体段落，确认谁对谁错
4. **修正文档**——在输出中更新错误结论
5. **追问根因**——为什么之前会得出错误结论？（链接不可靠？混淆功能？推理过度？）

### 常见错误防范

| 错误类型 | 表现 | 防范 |
|------|------|------|
| 功能混淆 | 把通用功能A的文档当专项功能B的引用 | 确认引用文档描述的场景和用户问的是同一个 |
| 过度推理 | 从"可选不删除"推断出"可浏览查看" | 用原文逐字对照，区分"能做什么"和"文档写了什么" |
| 故障当功能 | 把故障场景的副作用当成产品设计能力 | 确认是 by design 还是 side effect |
| 链接失效 | 用了打不开的第三方链接导致结论存疑 | 优先用官方域名，输出前验证链接 |

### 修正后必须做的事

1. 更新分析文档中的错误结论
2. 在文档中标注修正原因
3. 向用户说明"之前结论有误，已修正，原因是XXX"

## 质检清单

**Step 3 通过前提**（以下任一项不满足，Step 3 不能结束）：

- [ ] 截图已实际内嵌且渲染可见（逐张打开 markdown 文件确认，缺 1 张即不通过。**如果关键结论依赖控制台或文档的可视化证据而非纯文本推断，该证据必须有截图——不能仅靠文字描述**）
- [ ] 所有【待验证】项已在最终文档中回应，或在附录中显式标注为【待补充】。**【待补充】项目必须同时标注搜索范围和排除证据，不能只写'未确认'——如'在 X 文档和 Y KB 中搜索，未找到 Z 说明'**

**专项自检**：
- [ ] **1. 问题定义力**：S0 是否真正理解了用户问题的本质？如果跳过了问题讨论直接搜答案，输出的解答大概率答非所问。S0 的讨论充分性决定了整个 Skill 的质量。
- [ ] **2. 解答针对性**：给出的解答是否直接回应了用户的原始问题？每家厂商的解法是否都明确标注了"对原始问题的回答"？结论是否放在开头，先回答再展开？
- [ ] **3. 取舍理解力**：是否真正理解了每家为什么这么解——不是仅罗列"他们做了什么"，而是基于产品定位、目标客群、技术路线，分析"他们为什么选这种解法而非另一种、这个选择有什么收益和代价"？是否区分了"业界共识""差异化选择""潜在坑点"？
- [ ] **4. 设计指导力**：分析结果能否转化为可操作的设计建议——不是泛泛的"业界都这么做"，而是"我们该怎么做、为什么参考 A 厂商的做法而非 B 厂商"？
