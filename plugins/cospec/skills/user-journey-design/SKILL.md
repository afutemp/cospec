# 用户旅程设计

**Skill 标识**: `user-journey-design`

其他 skill 通过 `user-journey-design` 引用本 skill。

## 核心定位

把已通过评审、决定要做的用户需求，细化为可用于需求文档、方案设计、UI 生成、代码生成、测试设计和用户需求评审的用户旅程设计产物。

本 Skill 的核心不是一次性生成完整文档，而是通过"输入理解确认 → 方案设计共创 → 未来旅程共创 → 最终文档生成"的分阶段流程，帮助产品经理把需求想完整、让方案选型清晰、把客户的真实操作流程讲明白。

## 强制规范

### 1. 必须保持状态机工作

每一轮只能处于一个阶段：

1. Step 0：输入理解确认。
2. Step 1：方案设计共创。
3. Step 2：未来旅程共创。
4. Step 3：最终文档生成。

没有用户明确确认上一阶段，不得进入下一阶段。

如果用户要求"直接生成完整文档"，只能生成"未经确认草稿"，并在开头明确标注：

> 本文档未经过输入理解、方案选型和未来旅程确认，仅作为草稿，不应直接作为最终用户旅程设计文档。

### 2. 有输入物不等于已确认

用户提供需求说明、PRD、会议纪要、竞品分析或客户反馈，只能说明"有输入物"，不能等同于"AI 对输入物的理解已经被产品经理确认"。

首次输出必须先输出输入理解确认，除非用户明确要求跳过确认。

### 3. 不确定信息必须显式标注

对输入中没有明确依据的内容，使用"待验证假设"或"待确认事项"标注，不要补成确定事实。

### 4. 方案必须覆盖老客户升级影响

方案设计不能只顺着用户的原始想法走下去。必须主动检查：

- 是否影响老客户升级。
- 是否影响现网兼容。
- 是否改变用户已有操作习惯。
- 是否需要灰度、回滚、默认策略或配置继承。
- 是否存在竞品更优做法。
- 是否存在跨产品可参考方案。
- 是否存在异常、分支或边界场景。

## 输入要求

优先要求用户提供以下输入：

- 需求背景：业务目标、目标客户、场景、客户痛点、需求来源。
- 现状旅程：当前客户如何完成该任务，在哪些步骤发生断点。
- 竞品分析：具体竞品、适用场景、用户旅程、能力优劣势、赛道信息来源。
- 预期目标：客户价值、体验目标、效率目标、业务结果或竞争目标。
- 老客户信息：存量客户版本、升级路径、历史配置、兼容约束、是否允许中断业务。
- 约束条件：技术限制、权限限制、交付限制、数据限制、灰度/回滚要求。

如果输入缺失：

- 影响输入理解或方案选型时，先提出不超过 5 个关键澄清问题。
- 不影响当前阶段推进时，用"待验证假设"标注，并继续推进当前阶段。

## 工作流

### Step 0：输入理解确认

触发条件：用户提供需求材料，或用户要求进行用户旅程设计时。

允许输出内容：只输出输入理解，不输出方案候选，不输出未来旅程，不输出最终文档。

输入理解必须包含：

| 模块 | 需要说明的内容 |
| --- | --- |
| 目标客户理解 | 客户类型、关键角色、组织环境、使用场景 |
| 场景理解 | 业务场景、触发条件、客户要完成的任务 |
| 客户痛点理解 | 客户当前遇到的具体问题，而不是抽象描述 |
| 现状旅程断点 | 当前流程中断点发生在哪一步，造成什么影响 |
| 竞品能力与优劣势理解 | 已知竞品分别做了什么，各自优势和劣势是什么 |
| 老客户升级影响初判 | 是否涉及存量版本、配置继承、兼容、灰度或回滚 |
| 预期目标理解 | 客户价值、体验、效率、业务结果、竞争目标 |
| 待验证假设 | 输入中没有证据但可能影响方案的假设 |
| 关键不确定点 | 会影响后续方案选型的问题 |

输出必须以确认问题收尾：

> 请确认上述输入理解是否准确；如有偏差，请直接修正目标客户、场景、痛点、现状旅程、竞品分析、老客户升级影响或预期目标。确认后，我再进入方案设计共创。

### Step 1：方案设计共创

触发条件：仅在用户确认或修正 Step 0 输入理解后执行。

允许输出内容：输出 2-4 个方案候选、方案判断建议、推荐方案和确认问题。不得输出未来旅程正文，不得输出最终文档。

每个方案必须包含：

| 字段 | 说明 |
| --- | --- |
| 方案名称 | 用一句话说明方案方向 |
| 核心设计思路 | 为什么这样设计 |
| 满足的客户痛点 | 对应 Step 0 中已确认的痛点 |
| 关键能力或机制 | 方案依赖的关键产品能力 |
| 对现状旅程的改变 | 客户流程中哪些步骤会发生变化 |
| 老客户升级影响 | 对存量版本、配置继承、兼容、灰度或回滚的影响 |
| 逐竞品优劣势对比 | 必须对比具体竞品，而不是泛写"相比竞品有差异" |
| 主要风险和约束 | 技术、体验、交付、数据、兼容风险 |
| 推荐适用条件 | 什么场景适合采用该方案 |

#### 老客户升级影响评估

方案选型必须单独评估：

| 维度 | 需要判断的问题 |
| --- | --- |
| 兼容性 | 是否兼容老版本、老配置、老客户端或老数据 |
| 升级路径 | 老客户如何从旧流程迁移到新流程 |
| 配置继承 | 历史配置是否继承，是否需要迁移或重新配置 |
| 默认策略变化 | 默认行为是否变化，是否影响客户预期 |
| 业务中断 | 是否会暂停客户正在运行的业务 |
| 灰度与回滚 | 是否支持灰度发布、失败后回滚和异常恢复 |
| 客户沟通 | 是否需要提示、公告、管理员确认或迁移指导 |

#### 逐竞品优劣势对比

不能只写"相比竞品有差异"。必须对具体竞品逐一比较：

| 竞品 | 竞品做法 | 我方优势 | 我方劣势 | 风险 | 可借鉴点 | 待验证问题 |
| --- | --- | --- | --- | --- | --- | --- |
| 竞品 A |  |  |  |  |  |  |
| 竞品 B |  |  |  |  |  |  |

如果没有可靠竞品资料，必须标注"待验证"，不得编造竞品做法。

#### 方案判断建议

必须同时给出判断结论和判断依据：

| 判断维度 | 判断结论 | 判断依据 | 对方案选择的影响 |
| --- | --- | --- | --- |
| 客户价值强度 |  |  |  |
| 老客户升级影响 |  |  |  |
| 实现复杂度 |  |  |  |
| 对现有流程扰动 |  |  |  |
| 与具体竞品差异化程度 |  |  |  |
| 风险可控性 |  |  |  |
| 是否适合作为首版 |  |  |  |

输出必须以确认问题收尾：

> 请确认选择哪个方案作为后续未来旅程设计基础；也可以指定"采用方案 A 的主流程 + 方案 B 的某个机制"，或由产品经理自行给出新的方案。确认方案后，我再进入未来旅程共创。

### Step 2：未来旅程共创

触发条件：仅在用户确认方案后执行。

允许输出内容：输出未来旅程边界、主流程、分支流程、异常流程、体验与实现提醒、确认问题。不得输出最终文档。

#### 1. 旅程边界

- 使用场景：
- 起点：
- 终点：
- 成功标准：
- 不覆盖范围：

| 1 |  |  |  |  |  |  |

| 分支场景 | 触发条件 | 适用角色/对象 | 用户操作 | 系统处理 | 结束状态 |

#### 5. 体验与实现提醒

需补充：

- 哪些环节需要降低用户判断成本。
- 哪些环节需要明确解释系统判断依据。
- 哪些环节需要保留人工确认。
- 哪些数据需要沉淀，供后续推荐、审计、复盘或自动化使用。
- 哪些环节会影响老客户升级、兼容、灰度或回滚。

输出必须以确认问题收尾：

> 请确认该未来旅程是否符合目标客户的真实工作方式；如需调整，请指出要修改的阶段、角色、主流程、分支流程、异常流程或结束状态。确认后，我再生成最终用户旅程设计文档。

### Step 3：输出用户旅程设计文档

触发条件：仅在输入理解、方案选型和未来旅程都被用户确认后执行。

最终文档必须使用 Markdown，并包含以下 4 个一级模块：

1. 需求背景。
2. 方案设计。
3. 未来旅程。
4. 目标达成情况分析。

最终文档规范：

- 将输入理解转化为已确认背景，不保留未确认口吻。
- 将方案候选摘录为已确认方案。
- 未选方案只在"方案选型说明"中简要说明未选用原因。
- 如果产品经理自行给出方案，以产品经理方案为基础，并补充痛点、旅程变化、竞品差异、风险约束和目标达成逻辑。
- 将未来旅程写成客户视角的完整流程，而不是会议纪要。
- 待验证假设集中放在相关模块末尾，方便后续评审跟进。

文档模板见：[assets/templates/document-template.md](assets/templates/document-template.md)

## Acceptance Criteria

输出后必须逐项检查：

- [ ] 是否完成输入理解确认（Step 0 已被用户确认）
- [ ] 是否完成方案选型确认（Step 1 已被用户确认）
- [ ] 是否完成未来旅程确认（Step 2 已被用户确认）
- [ ] 是否包含分支流程和异常流程内容
- [ ] 是否评估老客户升级影响（兼容、灰度、回滚、配置继承）
- [ ] 是否逐竞品比较，而不是泛写"相比竞品"
- [ ] 是否说明竞品赛道信息可信度和待验证问题
- [ ] 是否对不确定信息标注为"待验证假设"
- [ ] 是否避免只写系统页面跳转，而体现了客户的真实工作流程
- [ ] 最终文档是否包含需求背景、方案设计、未来旅程、目标达成分析四个模块

## 输出风格

- 默认使用中文。
- 使用产品经理、研发、设计、测试都能共同理解的表达。
- 优先结构化表格和分阶段流程。
- 避免营销化表达，避免空泛形容词。
- 对需要产品经理确认的内容，用清晰的确认项收尾。
