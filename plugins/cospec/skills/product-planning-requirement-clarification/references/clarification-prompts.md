# Clarification Prompts

## First Round: Requirement Diagnosis

Use when product planning first submits a requirement.

```text
请基于我填写的【需求】，帮助我把需求背景想全面。

你的任务不是写需求文档，而是作为“需求澄清助手”，帮助我识别这个需求中最关键、最影响下游判断的信息缺口。请聚焦重点问题，不追求大而全。

我的需求如下：
【粘贴产品规划填写的需求模板内容】

请先完成第一轮需求诊断，按 5 个维度分析：

1. 场景
这个需求发生在什么用户场景？当前流程是什么？目标流程是什么？不解决会有什么影响？

2. 角色与对象
涉及哪些用户角色、系统角色或外部系统？他们操作的核心对象是什么？例如资源、数据、配置、策略、任务、日志、告警、接口等。

3. 流程与规则
用户或系统的关键动作是什么？有哪些触发条件、状态变化、权限规则、依赖关系或冲突规则？

4. 异常与边界
最需要提前想清楚的异常、误操作、失败、冲突、性能、兼容、安全或回滚场景是什么？本需求有哪些边界或不做事项？

5. 下游影响
这个需求会影响研发、测试、市场、服务分别哪些工作？哪些信息如果不确认，会导致后续返工或理解偏差？

请输出：
1. 你对需求的初步理解；
2. 已知事实；
3. 合理假设；
4. 关键缺口；
5. 不超过 5 个最重要追问。

追问要求：
- 每个问题必须具体、可回答；
- 每个问题说明“为什么要问”；
- 每个问题说明会影响研发、测试、市场或服务的哪类工作；
- 如果我暂时无法回答，请基于合理假设继续，但必须明确标记为“待确认假设”。
```

## Middle Rounds: Continue Clarification

Use after product planning answers one or more questions.

```text
请基于我刚才的回答，继续判断这个需求是否已经“想全面”。

请不要写需求文档，也不要发散新范围。请只做三件事：

1. 更新已确认事实、合理假设和待验证问题；
2. 判断当前还缺哪些会影响研发、测试、市场或服务判断的关键信息；
3. 如果仍需追问，请只提出不超过 5 个最重要问题，并说明每个问题的影响。

如果你认为信息已经足够支撑后续用户需求文档撰写，请明确说明“可以收敛”，并给出建议收敛理由。
```

## Convergence Round: Extract Deliverable

Use when product planning believes the requirement has been sufficiently clarified.

```text
请基于以上多轮对话，提取“想全面”阶段已经形成的核心结论，输出一份【需求背景澄清交付物】。

要求：
1. 不继续发散新范围；
2. 不编造对话中没有的信息；
3. 只提炼核心结论，表达要凝练、准确、可交付；
4. 如存在不确定信息，请明确标记为“待确认”；
5. 如存在前后不一致，请标记为“信息冲突”。

请按以下结构输出：

1. 需求概述
- 需求名称；
- 一句话需求；
- 目标用户；
- 核心场景；
- 要解决的主要问题；
- 期望达成的结果。

2. 需求全景结论
- 关键角色；
- 核心对象；
- 关键动作；
- 主要流程；
- 关键状态变化；
- 重要业务规则或约束。

3. 异常、边界与风险
- 需要重点考虑的异常场景；
- 需要明确的边界条件；
- 本需求暂不覆盖的内容；
- 可能影响实现、测试或交付的风险。

4. 下游影响
- 研发需要关注什么；
- 测试需要覆盖什么；
- 市场可以提炼什么价值；
- 服务需要准备什么交付和运维信息。

5. 事实、假设与待确认
- 已确认事实；
- 合理假设；
- 待确认问题。

最后，请用 3 到 5 条总结本需求最重要的共识，作为后续撰写用户需求文档和 Demo 输入的基础。
```
