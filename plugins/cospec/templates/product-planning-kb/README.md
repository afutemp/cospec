# 产品规划知识库模板

> 面向产品经理的产品规划知识库骨架。模板参考 `product-knowledge` 实例的小颗粒度拆分、统一元数据、版本标签、模块索引和跨文档追踪方式，并将重点从“完整需求规格”调整为“战略—机会—路线图—功能—验证”的规划闭环。

## 使用方式

1. 复制本目录并以产品名命名。
2. 将所有 `{{占位符}}` 替换为真实内容；无信息时保留 `待确认`，不要编造。
3. 一个主题一份文档，避免把所有规划放入单一大文件。
4. 每份文档保留 YAML 元数据，用于按产品、版本、状态、范围检索。
5. 功能规划文档统一使用 `03-功能规划/功能模板.md` 复制创建。
6. 重要结论需记录证据来源、假设和待验证项。

## 文档元数据约定

```yaml
---
name: 文档名称
description: 一句话描述本文覆盖的规划信息
source: 来源文档、访谈、数据或会议
product: 产品名称
version: 目标版本或 ALL
scope: DISCOVERY | MVP | V1.X | FULL
status: DRAFT | REVIEW | BASELINED | ARCHIVED
owner: 负责人
updated: YYYY-MM-DD
---
```

## 标签约定

- `[ASSUMPTION]`：尚未验证的关键假设
- `[EVIDENCE]`：有数据、访谈或客户案例支持
- `[DECISION]`：已形成的产品决策
- `[OPEN]`：待确认问题
- `[MVP]`：最小可验证范围
- `[NEW]`：新能力
- `[ENHANCED]`：已有能力增强
- `[OUT]`：明确不在当前范围
- `[BLOCKED]`：被依赖或决策阻塞

## 建议维护节奏

- 每周：更新需求池、证据、风险、实验结果。
- 每个里程碑：更新版本范围、变更映射、成功指标。
- 每次评审后：把结论写成 `[DECISION]`，把未决项写成 `[OPEN]`。
- 每次范围变化：同步修改路线图、功能文档和依赖风险，避免文档间口径不一致。
